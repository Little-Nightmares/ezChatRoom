# ChatRoom 团队分工方案

> 4 人团队，基于项目骨架文件结构，按角色划分负责模块。

---

## 一、团队角色总览

| 角色 | 代号 | 核心职责 | 负责目录 |
|------|------|---------|---------|
| **客户端全能王** | A | QML 界面开发 + C++ Bridge 桥接层 | `client/qml/`、`client/src/qmlbridge/` |
| **客户端后勤部长** | B | 核心逻辑 + 本地数据库 + 加密模块 | `client/src/core/`、`client/src/database/`、`client/src/crypto/` |
| **网络快递员** | C | 客户端网络 + 服务端网络通信 | `client/src/network/`、`server/src/core/` |
| **服务端管家** | D | 服务端业务逻辑 + 数据库 + 加密 | `server/src/database/`、`server/src/crypto/`、`server/src/models/` |

---

## 二、各角色详细分工

```

角色 A（界面） ←调用→ 角色 C（Network）提供的接口
角色 A（界面） ←展示→ 角色 D（服务端）返回的数据
```

### 角色 C：网络快递员（客户端网络 + 服务端网络）

**一句话描述**：负责数据在客户端和服务端之间的传输，是整个系统的"快递公司"。

#### 负责的文件清单

**客户端网络层（4 个类 × 3 文件 = 12 个文件）：**

| 文件 | 说明 | 优先级 |
|------|------|--------|
| `client/src/network/TcpClient.h/.cpp` | TCP 连接管理、数据收发 | ⭐⭐⭐ |
| `client/src/network/MessageHandler.h/.cpp` | 消息解析与分发 | ⭐⭐⭐ |
| `client/src/network/PacketBuilder.h/.cpp` | 协议包构建 | ⭐⭐⭐ |
| `client/src/network/HeartbeatManager.h/.cpp` | 心跳保活机制 | ⭐⭐ |

**服务端网络层（6 个类 × 3 文件 = 18 个文件）：**

| 文件 | 说明 | 优先级 |
|------|------|--------|
| `server/src/core/TcpServer.h/.cpp` | TCP 服务器、监听连接 | ⭐⭐⭐ |
| `server/src/core/ClientSession.h/.cpp` | 单个客户端会话管理 | ⭐⭐⭐ |
| `server/src/core/MessageRouter.h/.cpp` | 消息类型路由分发 | ⭐⭐⭐ |
| `server/src/core/ConnectionPool.h/.cpp` | 数据库连接池 | ⭐⭐ |
| `server/src/core/ThreadPool.h/.cpp` | 线程池管理 | ⭐⭐ |
| `server/src/core/SessionManager.h/.cpp` | 在线用户管理、心跳检测 | ⭐⭐⭐ |

**还需要负责：**
- `server/src/main.cpp` — 服务端入口（启动 TcpServer）
- **通信协议设计** — 定义消息包格式（包头、包体、消息类型码），这是客户端和服务端通信的"语言"，需要和角色 D 一起商量确定

#### 需要掌握的技术

| 技术 | 用在哪里 | 学习优先级 |
|------|---------|-----------|
| **QTcpSocket / QTcpServer** | 所有网络文件 | 最高 |
| **Qt 信号与槽** | 异步数据收发 | 最高 |
| **QDataStream** | 数据序列化/反序列化 | 最高 |
| **QThread / QtConcurrent** | ThreadPool、并发处理 | 高 |
| **自定义协议设计** | PacketBuilder、MessageRouter | 最高 |

#### 与其他角色的协作关系

```
角色 C（客户端网络） ←接收指令→ 角色 A（Bridge）发起连接/发送消息
角色 C（客户端网络） →传递数据→ 角色 B（Database）存储
角色 C（客户端网络） ←加密→ 角色 B（Crypto）加密数据
角色 C（服务端网络） →转发消息→ 角色 D（服务端业务）处理
角色 C ←协议约定→ 角色 D（服务端管家）共同定义通信协议
```

## 三、开发阶段与协作计划

### 阶段 1：基础搭建（第 1-2 周）

**目标**：各自搭建开发环境，跑通 Hello World。

| 角色 | 任务 | 产出 |
|------|------|------|
| **全员** | 安装 Qt 6.11.0 + Qt Creator + CMake | 能编译运行项目骨架 |
| A | 修改 `main.qml` 显示一个简单登录界面 | 客户端能看到登录页 |
| B | 实现 `DatabaseManager`，创建 SQLite 数据库和基础表 | 数据库能正常初始化 |
| C | 实现 `TcpServer.start()` 和 `TcpClient.connectToServer()` | 客户端能连上服务端 |
| D | 实现 `DatabasePool` + `User` 模型 + 建表 SQL | 服务端数据库就绪 |

**协作点**：
- C 和 D 需要一起商量**通信协议格式**（消息包怎么打包、消息类型码怎么定义）
- B 和 D 需要一起商量**数据库表结构**（哪些表、哪些字段）

### 阶段 2：通信链路打通（第 3-4 周）

**目标**：客户端发消息 → 服务端收到 → 服务端回消息 → 客户端收到。

| 角色 | 任务 | 产出 |
|------|------|------|
| C | 实现 `PacketBuilder`（打包）+ `MessageHandler`（解包）+ `MessageRouter`（路由） | 消息能正确打包和路由 |
| C | 实现 `ClientSession`（管理单个连接）+ `SessionManager`（管理所有在线用户） | 服务端能管理多个客户端连接 |
| B | 实现 `SessionManager`（客户端侧，管理登录状态） | 客户端能记录连接状态 |
| A | 实现 `AppCore`（Bridge 单例），把网络状态暴露给 QML | 界面能显示"已连接/未连接" |

**协作点**：
- C 定义好消息类型码后，通知 A 和 D
- A 需要调用 C 的接口来展示连接状态

### 阶段 3：用户系统（第 5-6 周）

**目标**：能注册、能登录、能显示用户信息。

| 角色 | 任务 | 产出 |
|------|------|------|
| D | 实现 `UserRepository`（注册、登录验证）+ `PasswordHasher`（密码哈希） | 服务端能验证用户 |
| D | 实现 `AuthHelper`（Token 生成与验证） | 登录成功后返回 Token |
| C | 在 `MessageRouter` 中添加登录/注册消息类型的处理 | 登录请求能路由到 D 的逻辑 |
| B | 实现 `UserDAO`（本地保存用户信息）+ `SettingsManager`（记住登录状态） | 客户端能记住用户 |
| A | 完善 `LoginPage.qml` + `RegisterPage.qml` + `UserController`（Bridge） | 用户能通过界面登录注册 |

**协作点**：
- D 实现好 Repository 后，C 负责在路由中调用
- C 把服务端返回的结果传给 B 存本地、传给 A 显示界面

### 阶段 4：好友系统（第 7-8 周）

**目标**：能搜索添加好友、显示好友列表、处理好友请求。

| 角色 | 任务 | 产出 |
|------|------|------|
| D | 实现 `FriendRepository` + `FriendRequest` 模型 | 服务端能管理好友关系 |
| C | 添加好友相关消息类型的路由处理 | 好友请求能正确转发 |
| B | 实现 `UserDAO`（联系人缓存） | 本地能缓存好友列表 |
| A | 实现 `FriendRequestPage.qml` + `ContactItem.qml` + `FriendController` | 界面能展示好友列表和请求 |

### 阶段 5：聊天功能（第 9-10 周）

**目标**：能发送文字消息、显示聊天记录。

| 角色 | 任务 | 产出 |
|------|------|------|
| D | 实现 `MessageRepository` + `OfflineMessageManager` | 服务端能存储和转发消息 |
| C | 实现消息转发的完整链路 | 消息能从发送方到达接收方 |
| B | 实现 `MessageDAO` + `ConversationDAO` | 本地能保存聊天记录 |
| B | 实现 `UserModel` + `MessageModel` + `ConversationModel` | 为 QML 提供列表数据 |
| A | 实现 `ChatPage.qml` + `MessageBubble.qml` + `InputBar.qml` + `ConversationItem.qml` + `ChatController` | 完整聊天界面 |

### 阶段 6：加密功能（第 11 周）

**目标**：消息传输加密，服务端无法解密。

| 角色 | 任务 | 产出 |
|------|------|------|
| B | 实现 `AESEncryptor` + `RSAEncryptor` + `KeyManager`（客户端侧） | 客户端能加密/解密消息 |
| D | 实现 `AESEncryptor` + `RSAEncryptor`（服务端侧，仅密钥交换用） | 服务端能协助密钥交换 |
| C | 在消息收发流程中集成加密/解密 | 传输链路加密完成 |

## 五、Git 协作建议

### 分支策略

```
main（主分支，稳定版本）
  ├── dev（开发分支，日常合并到这里）
  │     ├── feature/login（登录功能）
  │     ├── feature/chat（聊天功能）
  │     ├── feature/friend（好友功能）
  │     └── feature/crypto（加密功能）
  └── release（发布分支）
```

### 协作规范

1. **每个人在自己的分支上开发**，不要直接改 main 或 dev
2. **改代码前先 pull**，避免冲突
3. **每天至少 commit 一次**，写清楚提交信息
4. **改完一个功能后提 Pull Request**，让至少 1 个人 review
5. **冲突了自己解决不了就及时沟通**，不要拖着

### 提交信息格式

```
[模块] 简短描述

示例：
[network] 实现 TcpClient 连接服务端功能
[qml] 完成 LoginPage 登录界面布局
[database] 创建 SQLite 数据库和用户表
[server] 实现 TcpServer 监听 6667 端口
```

## 六、沟通机制

| 会议 | 频率 | 参与人 | 内容 |
|------|------|--------|------|
| 每日站会 | 每天 | 全员 | 昨天做了什么、今天要做什么、有没有卡住 |
| 技术对接会 | 每周 2 次 | 相关角色 | 接口定义、协议约定、联调 |
| 阶段评审 | 每阶段结束 | 全员 | 演示成果、发现问题、调整计划 |

### 关键对接关系

| 对接方 | 讨论内容 | 时机 |
|--------|---------|------|
| **C ↔ D** | 通信协议格式、消息类型码 | 阶段 1 开始前 |
| **B ↔ D** | 数据库表结构设计 | 阶段 1 开始前 |
| **A ↔ C** | 网络接口怎么暴露给 QML | 阶段 2 |
| **A ↔ B** | 数据模型接口（Model） | 阶段 3-5 |
| **B ↔ C** | 加密集成到网络层 | 阶段 6 |

## 七、风险与注意事项

| 风险 | 影响 | 应对方案 |
|------|------|---------|
| 通信协议设计不合理 | 后期大量返工 | C 和 D 在阶段 1 花足够时间设计好协议 |
| 接口不兼容 | A 调不到 B/C 的接口 | 每个阶段开始前先定义好接口头文件 |
| Git 冲突太多 | 浪费时间 | 各自负责不同目录，尽量减少改同一个文件 |
| 某人进度落后 | 整体延期 | 每日站会及时发现问题，互相帮助 |
| 加密太复杂做不出来 | 功能缺失 | 先做明文通信，最后再加加密 |
